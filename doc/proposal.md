# ReviewPilot — AI PR Review 助手需求文档

## 1. 项目概述

### 1.1 项目背景

代码评审（Code Review）是保证软件质量的核心环节，但在企业团队中面临以下痛点：

- 高级开发者的评审带宽有限，成为交付瓶颈
- 新人提交的 PR 需要反复修改，沟通成本高
- 评审标准因人而异，缺乏一致性
- 安全漏洞和性能问题容易被人工忽略

**ReviewPilot** 是一个 AI 驱动的 GitHub PR 评审助手，以 GitHub Bot + 轻量 Web 应用的双形态交付，自动获取 PR 变更并智能分析，帮助团队提升评审效率与质量。

### 1.2 产品定位

| 维度 | 说明 |
|------|------|
| 产品形态 | GitHub App（Bot）+ 轻量 Web Dashboard |
| 目标用户 | 企业开发团队（多仓库、多用户） |
| 核心价值 | 自动发现 PR 中的安全、逻辑、性能、风格问题，输出行级评论 + 总结报告 |
| 差异化 | 保守可配置的误报控制、Python/Java 深度分析、多仓库统一管理 |

## 2. 用户角色与使用场景

### 2.1 用户角色

| 角色 | 说明 |
|------|------|
| PR 提交者 | 提交 PR 后收到 AI 初审反馈，在评审前自助修复问题 |
| PR 评审者 | 参考 AI 分析结果，快速定位高风险代码，提升评审效率 |
| 团队管理员 | 配置仓库的评审规则、灵敏度、忽略规则，查看团队评审数据 |

### 2.2 核心使用场景

**场景 A：PR 自动评审**
> 开发者提交 PR → GitHub Webhook 触发 → ReviewPilot 拉取 diff 及上下文 → AI 分析 → 在 PR 中创建行级评论 + 总结评论

**场景 B：手动触发评审**
> 评审者在 PR 中评论 `/review` → ReviewPilot 解析命令 → 执行分析 → 追加评论

**场景 C：Dashboard 查看**
> 团队管理员打开 Web Dashboard → 查看各仓库的评审记录、问题分布、团队活跃度统计

**场景 D：评审规则配置**
> 管理员在 Web 端为某个仓库配置：关闭风格检查、提升安全规则灵敏度、添加自定义忽略规则

## 3. 功能需求

### 3.1 GitHub Bot 模块（R1）

#### R1.1 PR 事件监听
- 监听 PR `opened`、`reopened`、`synchronize`（新提交推送）事件
- 事件到达后自动触发分析流程
- 支持按仓库配置是否开启自动评审

#### R1.2 手动命令触发
- 监听 PR 评论 `created` 事件
- 解析 `/review` 命令（可带参数，如 `/review focus:security`）
- 仅仓库协作者或特定权限用户可触发

#### R1.3 行级评论（Review Comment）
- 对具体代码行创建 Review Comment，关联到 PR 的 `diff_hunk`
- 每条评论标注严重级别（Critical / Warning / Suggestion）
- 包含问题描述 + 修复建议代码片段

#### R1.4 总结评论（Summary Comment）
- 在 PR 对话中创建一条总体评审总结
- 包含：变更概述、问题统计（按严重级别）、高风险文件列表
- 如无问题则给出通过标记

#### R1.5 多仓库支持
- 一个 GitHub App 安装可授权多个仓库
- 每个仓库独立配置，互不干扰
- Bot 安装流程简洁（标准 GitHub App 安装流程）

### 3.2 AI 分析引擎（R2）

#### R2.1 PR 变更总结
- 理解 PR 的整体目的和变更范围
- 生成简洁的变更摘要（中文/英文可配置）
- 识别变更涉及的功能模块

#### R2.2 风险代码识别

| 类别 | 检查项示例 |
|------|-----------|
| 安全 | SQL 注入、XSS、敏感信息泄露、权限绕过、不安全的反序列化 |
| 逻辑 | 空指针/None 引用、边界条件错误、资源泄漏、死锁风险 |
| 性能 | N+1 查询、不必要的循环、内存泄漏、阻塞 IO |
| 最佳实践 | 异常处理不当、缺少输入校验、硬编码配置、可维护性问题 |
| 代码风格 | 命名规范、函数过长、圈复杂度高、重复代码 |

#### R2.3 Review 建议生成
- 对每个发现的问题提供具体的修复建议
- 建议包含修复代码示例（diff 格式）
- 引用相关的最佳实践或规范文档

#### R2.4 语言支持策略

| 语言 | 分析深度 | 分析手段 |
|------|---------|---------|
| Python | 深度 | AST 解析 + AI 上下文分析 |
| Java | 深度 | AST 解析 + AI 上下文分析 |
| JavaScript/TypeScript | 通用 | Diff 文本 + AI 上下文分析 |
| 其他语言 | 通用 | Diff 文本 + AI 上下文分析 |

深度分析指结合 AST（抽象语法树）进行结构化分析，可精确识别函数边界、变量作用域、控制流等。

#### R2.5 上下文获取策略
- **核心上下文**：PR 的完整 diff（包含周围代码行）
- **增强上下文**：变更文件的全文件内容（按需获取）
- **项目上下文**：项目结构、依赖声明、配置文件（用于安全分析和最佳实践判断）
- 上下文窗口管理：优先保证核心上下文完整，增强和项目上下文按 token 预算动态加载

### 3.3 Web Dashboard 模块（R3）

#### R3.1 评审记录管理
- 评审记录列表（按时间、仓库、PR 状态筛选）
- 单条评审详情页（问题列表 + 代码上下文展示）
- 评审状态追踪（待处理 / 已确认 / 已忽略）

#### R3.2 统计概览
- 团队评审数据：PR 数量、问题总数、平均问题数/PR
- 问题分布：按严重级别、按类别（安全/逻辑/性能等）
- 趋势图：问题发现率随时间变化
- 仓库维度对比

#### R3.3 配置管理
- 仓库列表管理（关联 GitHub 仓库）
- 评审规则开关（启用/禁用特定检查类别）
- 灵敏度配置（高/中/低，控制报告阈值）
- 忽略规则管理（按文件模式、规则 ID 忽略）

#### R3.4 PR 详情增强视图
- Web 端查看 PR 的完整分析结果
- 代码 diff 高亮 + 问题标注
- 比 GitHub 原生界面更丰富的分析信息展示

### 3.4 误报与漏报控制（R4）

#### R4.1 默认保守策略
- 默认仅报告高置信度（≥80%）的问题
- 中置信度问题标记为 Suggestion（建议）
- 可配置各检查类别的独立灵敏度

#### R4.2 分级报告

| 级别 | 标识 | 说明 | 默认行为 |
|------|------|------|---------|
| Critical | 红色 | 高概率导致安全漏洞或线上故障 | 必须报告 |
| Warning | 橙色 | 潜在风险，中等概率 | 报告（可关闭） |
| Suggestion | 灰色 | 改进建议，低概率 | 默认关闭，可开启 |

#### R4.3 反馈学习
- 用户在 PR 中对 AI 评论的 Reply/Resolve 操作可记录为反馈信号
- 未来可基于反馈数据优化 prompt 和规则（列为扩展方向）

## 4. 非功能需求

### 4.1 性能

| 指标 | 目标 |
|------|------|
| 分析响应时间 | PR ≤100 行 diff：< 30 秒；≤500 行 diff：< 90 秒 |
| Web 页面加载 | 首屏 < 2 秒 |
| 并发分析 | 支持 3-5 个 PR 同时分析 |

### 4.2 安全性

- GitHub App 私钥安全存储（环境变量或密钥管理服务）
- Web Dashboard 用户认证（GitHub OAuth 登录）
- API 访问控制（仅授权用户可查看对应仓库的评审数据）
- 不在日志中记录完整的源代码内容

### 4.3 可扩展性

- AI 模型提供商可切换（OpenAI-compatible API 抽象层）
- 数据库从 SQLite 可平滑迁移到 MySQL
- 分析引擎支持添加新的语言分析器
- 支持通过消息队列（Redis/RabbitMQ）异步化分析任务

### 4.4 可用性

- 明确的错误提示（PR 过大无法分析、API 调用失败等）
- Bot 未响应时的降级策略（提示用户稍后重试）
- 首次安装引导流程（Web 端配置指引）

## 5. 技术架构

### 5.1 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 后端框架 | FastAPI (Python 3.10+) | 异步支持、自动 OpenAPI 文档 |
| 数据库 | SQLite（开发）→ MySQL（生产） | 轻量起步、SQLAlchemy ORM 保证迁移平滑 |
| 消息队列 | 可选：Redis / RabbitMQ | 异步分析任务，初期可同步处理 |
| AI API | OpenAI-compatible API | 支持任意兼容 OpenAI 接口的模型 |
| GitHub 集成 | PyGithub + GitHub Webhooks | PR 操作 + 事件接收 |
| 前端 | Jinja2 模板 + HTMX + Tailwind CSS | 轻量、无需重型前端框架 |
| AST 解析 | Python `ast` 标准库 / Java `javalang` | 结构化代码分析 |
| 部署 | Docker + Uvicorn | 容器化部署 |

### 5.3 数据模型（核心实体）

```
Repository: id, github_id, name, owner, config (JSON), created_at
PullRequest: id, github_pr_id, repository_id, number, title, status, created_at

```

## 6. AI 模型设计

### 6.1 模型选择

核心要求：**OpenAI-compatible API**，可灵活切换模型。

推荐方案：
- **默认模型**：`gpt-4o-mini`（成本低、速度快，适合初筛）或 `deepseek-v3`（性价比高）
- **深度分析模型**：`gpt-4o` 或 `claude-sonnet-4-6`（处理复杂逻辑问题时使用）
- **扩展设计**：Provider 抽象层，通过配置切换模型，支持 `openai`、`anthropic`、`deepseek` 等

### 6.2 Prompt 策略

- **系统 Prompt**：定义评审角色、评审标准、输出格式约束
- **用户 Prompt**：包含 PR 元信息 + diff 内容 + 文件上下文
- **分阶段调用**（扩展方向）：
  - 阶段 1：变更总结（快速、低 token）
  - 阶段 2：安全/逻辑风险扫描（并行、深度分析）
  - 阶段 3：风格/最佳实践检查（轻量）

### 6.3 上下文获取方式

| 级别 | 内容 | Token 消耗 | 何时使用 |
|------|------|-----------|---------|
| L0 | PR diff only | 低 | 初步扫描，快速判断 |
| L1 | Diff + 周围 50 行 | 中 | 需要理解上下文时 |
| L2 | Diff + 完整文件 | 高 | 发现可疑代码需要全文件理解 |
| L3 | Diff + 完整文件 + 关联文件 | 很高 | 跨文件影响分析（如 API 变更） |

默认策略：L1 作为标准配置，L2 在发现可疑模式时按需加载。

## 8. 未来扩展方向（不纳入本期开发）

### 8.1 短期扩展（v1.1）
- **Review 自定义规则引擎**：用户可定义自己的检查规则（正则/pattern）
- **评审结果通知**：Slack/企业微信/钉钉通知集成
- **Ignore 标记系统**：在代码中添加 `// reviewpilot:ignore` 注释来抑制特定检查

### 8.2 中期扩展（v1.2）
- **分阶段 AI 分析**：变更总结 → 分类扫描 → 深度分析的流水线
- **反馈闭环**：基于用户 Resolve/Reply 行为优化 prompt 和规则权重
- **评审模板**：不同仓库类型（库/服务/前端）使用不同评审模板
- **更多语言深度分析**：Go、Rust

### 8.3 长期扩展（v2.0）
- **PR 影响范围分析**：基于调用链分析变更的影响范围
- **自动化修复**：对简单问题直接创建 fix PR
- **团队评审质量度量**：评审覆盖率、评审响应时间、问题修复率等指标
- **代码知识图谱**：构建项目级代码知识库，提升跨文件分析准确度

## 9. 关键设计决策

### 9.1 为什么选择 GitHub Bot + Web 双形态？

- **Bot**：融入开发者现有工作流（GitHub PR 页面），不需要切换工具
- **Web**：提供 GitHub 原生界面无法提供的聚合视图（跨仓库统计、配置管理、增强详情）

### 9.2 为什么 AI 模型用 OpenAI-compatible API 抽象？

- 不被单一模型绑定，团队可根据成本和效果自由切换
- 市场上大量模型（DeepSeek、Qwen、Claude 等）都支持此接口格式
- 符合比赛中「模型选择设计思路」的评审加分点

### 9.3 为什么默认保守策略？

- 企业团队场景下，误报（False Positive）比漏报更伤害信任
- 过多低质量建议会导致「狼来了」效应，评审者直接忽略所有 AI 评论
- 保守策略建立信任后，用户可自行调高灵敏度

## 10. 成功标准

- [ ] 对任意 GitHub 仓库的 PR，能在 30 秒内返回分析结果并评论到 PR
- [ ] Python/Java PR 的安全问题检出率 ≥ 70%（以 Benchmark 测试集为准）
- [ ] 误报率（用户标记为 Resolved/忽略的 issue 占比）≤ 30%
- [ ] Web Dashboard 能正确展示多仓库的统计数据和评审记录
- [ ] 完整 Docker 部署方案，一篇 README 即可拉起全套服务
- [ ] 开发周期内保持持续 PR 提交，commit 时间戳分布合理
